Scikit-Learn Logistic Regression

  • Sat 20 August 2016
  • ·
  • scikit-learn

  • python, scikit-learn, regression, logistic regression

The below provides a guide to implementing Logistic Regression using scikit-learn, a Python package specifically designed for the implementation of machine learning.

With the regression techniques covered in 'Scikit-learn Regression', the target variable being predicted is assumed to follow a Normal Distribution, with an infinite range of values on a continuous scale. In some cases, however, the target variable has a limited range of discrete values (e.g. Customer is Good Credit Risk or is Bad Credit Risk), and so this assumption is invalid.

We will now examine such an example, and how predictions can be generated using logistic regression.

Data Preparation

We use German Credit Data provided by the UCI Machine Lerning Repository (Lichman, 2013). Here we are seeking to predict whether the customer is a good or a bad credit risk.

The Class Imbalance Issue

There is a large imbalance in the classes. The number of 'good credit risks' is more than double the number of 'bad credit risks':

In [48]:
print "Number of bad credit risk: " + str((CreditData['CreditRisk'] == 2).sum())
print "Number of good credit risk: " + str((CreditData['CreditRisk'] == 1).sum())

Number of bad credit risk: 300
Number of good credit risk: 700

Research by Crone and Finlay (2012), examining consumer repayment data, suggested that such an imbalance does not affect the reliability of logistic regression techniques. Hence, the data will not be balanced. If we had used a decision tree approach instead, then Crone and Finlay's data (2012) suggested there would be an issue. In this case, they advocate oversampling the bad credit risk data. The paper can be found here.

Feature Engineering

Our features consist of a mix of numerical (Duration, CreditAmount), ordinal (CheckingAccount, CreditHistory) and nominal (Purpose) data. We need to convert this into numerical data.

CheckingAccount

Considering CheckingAccount, of particular interest will be whether the customer is overdrawn - i.e. in category A11. Hence, we create dummy variable Overdrawn that equals 1 if customer is in category A11 and 0 otherwise:

In [49]:

#Convert True to 1 and False to 0:
CreditData['Overdrawn'] = (IsOverdrawn).apply(lambda x: 1 if x else 0)

Purpose

Purpose is a nominal variable, comprising of different items, e.g. car (new) or retraining. We will split this up into a series of dummy variables where 1 indicates it is of a particular purpose and 0 indicates that it is not. To prevent multicollinearity, no dummy variable is provided for others. Hence, when all dummy variables values equal to zero, the purpose of the credit is others:

    CreditData[purpose] = (IsPurpose).apply(lambda x: 1 if x else 0)

Evaluating the Model

Classification Accuracy

A simplistic approach would be to use classification accuracy:

$ \frac{\#Correctly Classified}{\#Test Set} $

This can be done using the function .score():

In [60]:
reg_1.score(x_test, y_test)
Out[60]:
0.71999999999999997

This is better than just random selection (which would lead to an accuracy of 0.5). An alternative would be a majority class prediction - we always predict the more likely value (predict always good):

In [61]:
#Number in majority class/Total:
len(y_test[y_test == -1])/float(len(y_test))
Out[61]:

Our logistic regression model is not much of an improvement over majority class prediction. However, we have used a very simple model for credit risk prediction.

Precision and Recall

The provided metadata states that the cost of failing to predict a bad credit risk (False Negative) is five times the cost associated with incorrectly predicting a bad credit risk (False Positive). Hence, we are particularly concerned about the recall metric, which declines with a higher False Negative Rate:

$ Recall = \frac{\#TruePositive}{\#TruePositive + \#FalseNegative} $

In [62]:
from sklearn.metrics import recall_score

recall_score(y_true = y_test, #True labels
             y_pred = reg_1.predict(x_test) #Predictions
            )
Out[62]:
0.35135135135135137

Our classifier detected only 1/3 of the Bad Credit Risks in the test dataset.

For reference, we also inculde the precision metric, which declines with a higher False Positive Rate:

$ Precision = \frac{\#TruePositive}{\#TruePositive + \#FalsePositive} $

In [63]:
from sklearn.metrics import precision_score

precision_score(y_true = y_test, #True labels
             y_pred = reg_1.predict(x_test) #Predictions
            )
Out[63]:

Just over half of the bad credit risks indicated by our classifier were actually bad credit risks.

Adjusting the Threshold

Currently, our classifier will predict a bad credit risk if the predicted probability of a bad credit risk exceeds 0.5. However, we can see above that this is resulting in a poor recall metric. One approach would be to go back and change our original model to improve the classifier. Another is to decrease the threshold from 0.5, so that there are less False Negatives (and also more False Positives). We can illustrate the impact of this with a precision-recall curve:

In [64]:

Note that the perfect classifier will have a perfect precision (equal to one), as we increase the recall. We can thus compare different classifiers using the above precision-recall, and comparing which is closest to the perfect classifier.

For example, the test data suggests that if we reduce the threshold to 0.16, then over 90% of bad credit risk customers will be correctly detected. However, precision will be less than 40%:

Note that, in practice: if we were to use the above precision-recall curve to choose between classifiers and/or to select a threshold that provided the most suitable trade-off between precision and recall, we would use a validation dataset that was distinct from the test dataset. This would ensure that the model had not been optimised for the test dataset, and so the test error would be a more reliable estimate for the true error.

An alternative to this hold-out approach is cross-validation, which is more computationally intensive, but also means more data available for learning the model, and removes dependence on a train/validation data split.

Coefficient Path

With the linear regression models, with had a tuning parameter alpha used to control the degree of regularisation. We also used this parameter in the lasso regression to find those most important features that are most robust to increasing the tuning parameter.

With logistic regression in scikit-learn we have a similar parameter C, which has a similar purpose to alpha, except the smaller C is, the more regularisation.

The application of the lasso regression here is for illustrative purposes only. In practice, before applying a lasso regression, we would need to standardise the features such that they were on the same scale.

As with lasso regression, we can draw a coefficient path:

#Setting an array of c values to examine:
cs = np.array([0.0001, 0.001, 0.01, 0.1, 1])

#Applying a lasso aproach to select features - hence penaly l1:
log_reg = linear_model.LogisticRegression(C=1.0, penalty='l1', tol=1e-6)

coefs_ = []

for c in cs:
    log_reg.set_params(C=c) #update c value
    log_reg.fit(x_train, y_train) #fit model
    coefs_.append(log_reg.coef_.ravel().copy()) #flattens and appends the array 

coefs_ = np.array(coefs_) #Convert list to array

#Plot coefficient values against C:

The most robust feature to increasing C is ScaledCreditHistory. The above coefficient path also suggests that being overdrawn raises credit risk and borrowing to fund a used car reduces credit risk (holding everything else fixed).

Strangely, the poorer the credit history, the lower the probability of a bad credit risk. As we are using a very simple model, with few included features, this odd result could be due to omitted variable bias, with a poor credit history acting as a proxy for an excluded variable that improves credit risk. For example, perhaps someone with a poorer credit history might be more reluctant to borrow, unless they are sure they can pay back the money. Alternatively, the results may be being skewed by the dominance of A32 in the dataset (over 50% of customers):
